# Runtime Topology Consumers Audit

## Summary

Runtime-facing sector systems already consume `SectorTopologyMap` as validated, closed, Doom-like topology. The central contract is: topology IDs are stable and unique, linedefs have valid endpoint vertices and at least one sidedef, sidedefs point back to their linedef slot and sector, each sector's sidedefs extract into one outer loop plus optional holes, and no invalid line intersections/overlaps exist.

Most runtime consumers should be able to keep consuming a derived `SectorTopologyMap` unchanged if a future permissive authoring graph is added. The main requirement is that derivation produces the same valid topology contract before mesh generation, collision, lightmap layout/bake, preview rendering, and 3D surface editing run.

## Runtime Topology Contract Today

- `SectorTopologyMap` in `sources/sector_demo/SectorTopologyMap.h` is the shared runtime/editor topology container: textures, vertices, linedefs, sidedefs, sectors, static lights, preview settings, sky settings, directional light, lightmap settings, and baked lightmap metadata.
- `SectorTopologyTypes.h` defines the runtime geometry pieces. Linedefs are directed from start to end; front/back sidedefs occupy linedef slots; sidedefs reference a sector; sectors store heights, textures, `ceilingSky`, UVs, decals, ambient lighting, and default wall settings.
- `ValidateSectorTopologyMap()` in `SectorTopologyValidation.cpp` enforces unique positive IDs, valid linedef endpoints, no duplicated physical linedefs, valid sidedef slots, no sidedef-less linedefs, finite static-light values, no invalid intersections/overlaps, and extractable sector loops.
- `ExtractSectorTopologyLoops()` requires each sector to have sidedefs, each directed sidedef to match its linedef slot, each boundary vertex to have exactly one incoming and one outgoing edge, and extractable closed loops with valid orientation. Runtime consumers call this rather than accepting loose/open lines.
- `BuildSectorTopologyIndexes()` in `SectorTopologyMap.cpp` is transient lookup data only. It retains duplicates for diagnostics; runtime consumers generally use the validated map and direct find helpers.

## Mesh Generation Consumers

- `BuildSectorGeneratedGeometry()` in `SectorGeneratedGeometry.cpp` is the main derived-geometry builder. It validates the whole map first, rejects maps with no sectors, checks finite geometry/material values, extracts loops per sector, and emits `SectorGeneratedSurface` records.
- Floor and ceiling geometry are generated by `BuildTopologyFlatSurface()`. It converts extracted loops into an earcut polygon, looks up vertices by ID, triangulates with `mapbox::earcut`, computes chart dimensions, applies sector UV/decal settings, and stores `topologySectorId` in the surface ref.
- Normal ceilings are skipped when `SectorTopologySector::ceilingSky` is true. That means sky status is geometry-affecting before meshes and lightmap layout ever see the surfaces.
- Wall geometry is generated by iterating `map.sideDefs`. One-sided linedefs produce full `Wall` surfaces. Two-sided linedefs compare facing and opposite sector heights to produce `LowerWall` and `UpperWall` strips.
- Sky-sky portals are handled in `BuildSectorGeneratedGeometry()`: if both adjacent sectors have `ceilingSky`, upper wall generation is suppressed between them.
- Middle textures are generated by iterating two-sided linedefs after regular walls. If either side has a middle texture, it emits alpha-tested `Middle` surfaces for both directions, using sidedef/sector IDs for selection and lightmap identity. These middle surfaces receive baked light but are not treated as normal wall strips.
- `BuildSectorMeshes()` in `SectorMeshBuilder.cpp` calls `BuildSectorGeneratedGeometry()` and batches generated surfaces by texture/decal/lightmap flags. It does not inspect topology loops itself.
- `SectorMeshPreview::RebuildRendererResources()` in `SectorMeshPreview.cpp` calls `BuildSectorGeneratedGeometry()` before requesting textures, sky meshes, lightmap layout, and GPU meshes. Missing/pending textures fall back through asset handles/default material at draw time.

## Collision Consumers

- `SectorCollisionWorld::BuildFromTopology()` in `SectorCollisionWorld.cpp` validates the map, sorts sectors by ID, extracts loops for each sector, converts authored floor/ceiling heights to world units, and stores outer/hole loops in world XZ coordinates.
- Collision edges are derived from extracted loop edges. One-sided linedefs become `BlockingWall`; two-sided linedefs become `Portal` edges with `neighborSectorId` from the opposite sidedef's sector.
- `SectorTopologyLineDefFlags::blocksPlayer` is consumed by collision portal logic. A two-sided portal can still block player movement if the linedef flag is set.
- Gameplay movement and sector lookup operate entirely on the built `SectorCollisionWorld`: `FindSectorContainingPoint()`, `FindSectorContainingPointPreferCurrent()`, `FindSectorForPlayerFootprint()`, and `ResolveMovement()` use closed collision loops, portal neighbors, step height, ceiling height, and player radius. They do not consume render triangles.

## Lightmap Consumers

- `BuildSectorLightmapLayout()` in `SectorLightmap.cpp` calls `BuildSectorGeneratedGeometry()` and packs one chart per generated lightmap-receiving surface. The layout depends on generated surface order and chart dimensions.
- `BakeSectorLightmap()` rebuilds generated geometry from the map, verifies the supplied layout has the same surface count, builds bake triangles, creates a BVH, rasterizes chart texels, evaluates static point lights, optional directional light, AO, and indirect bounce, then exports the atlas.
- Static point lights come from `map.staticLights` and are converted from authoring units to world space by `MakeWorldSpaceLight()`.
- Directional light settings come from `map.directionalLight`, are normalized, and affect direct lighting. Directional light is applied only to surfaces owned by sky sectors via `IsSkyOwnedLightmapSurface()`.
- `BuildBakeTriangles()` skips surfaces for occlusion when `CastsLightmapOcclusion(surface)` is false. Current behavior keeps middle textures as receivers but not baked shadow occluders.
- `IsSameLogicalSectorLightmapSurface()` uses topology sector/linedef/sidedef IDs to ignore self hits for floors, ceilings, walls, lower walls, and upper walls. Middle surfaces deliberately do not share a logical-surface identity there.
- `ComputeSectorLightmapSourceHash()` records bake constants/settings, directional light, coordinate subdivision, referenced texture definitions, vertices, linedefs, sidedefs, sectors, `ceilingSky`, sector ambient values, wall parts, and static lights. It intentionally hashes geometry/bake inputs, not visual-only preview sky settings.

## Gameplay Preview Consumers

- `SectorEditor::TryEnterPreview3D()` in `SectorEditor.cpp` rebuilds preview renderer resources from `state.topologyMap`, then calls `RebuildSectorCollisionWorld()` for gameplay collision. If preview mesh generation fails, 3D mode is not entered.
- `RebuildSectorEditorCollisionWorld()` in `SectorEditorPreviewActions.cpp` builds `state.sectorCollisionWorld` from `state.topologyMap`, stores a warning on failure, and refreshes gameplay vertical context when needed.
- `UpdateSectorEditorGameplayPreview()` uses the already-built collision world for sector lookup, horizontal movement resolution, floor/ceiling vertical physics, and current-sector tracking. Visual step smoothing, headbob, and landing dip are applied after collision state and do not feed back into physics.
- Freefly preview mode consumes only the renderer pose and generated preview mesh. Gameplay mode additionally depends on successful `SectorCollisionWorld` construction.

## 3D Selection / Inspector Consumers

- `PickSectorGeneratedGeometry()` in `SectorGeneratedGeometry.cpp` raycasts generated triangles and returns a `SectorGeneratedSurfaceRef` carrying topology sector, linedef, sidedef, and side IDs.
- `SectorEditor::PickSectorSurface3D()` converts generated refs to editor refs using `ToEditorSurfaceRef()` and stores hover hit data.
- `SectorEditor::UpdatePreview3DSelection()` selects hovered generated surfaces on left click when the preview UI allows it.
- `SectorEditor::DrawPreviewSurfaceHighlights()` highlights selected/hovered surfaces by comparing the selected ref against each generated surface ref in `preview.RenderedGeometry()`.
- `SectorEditor::SelectSurface3D()` maps wall-like surfaces to sidedef material targets and flat surfaces to sector material targets. It calls `SelectTopologySideDef()` or `SelectTopologySector()`, so the selected generated surface must still correspond to live topology IDs.
- `SectorEditor::IsValidSurfaceRef()` revalidates selected refs against `state.topologyMap`. Wall refs require an existing sidedef whose linedef ID and side match; middle refs additionally require `IsTopologyMiddleEligible()`. Flat refs require an existing sector.

## Validation Assumptions

- Mesh generation and collision explicitly call `ValidateSectorTopologyMap()` before extraction. They still include local defensive checks for missing lookups and invalid spans, but their intended input is already-valid topology.
- Lightmap layout and bake rely on `BuildSectorGeneratedGeometry()`, so their topology validation is inherited from generated geometry.
- 3D picking uses generated geometry produced during preview rebuild. It assumes generated surface refs remain meaningful until the preview is rebuilt or the selection is cleared.
- Runtime systems do not tolerate loose authoring lines in `SectorTopologyMap`: linedefs with no sidedefs, open boundaries, duplicated/crossing lines, missing opposite sidedefs, or non-extractable sectors fail validation or loop extraction before runtime use.

## Which Consumers Could Use A Derived Topology Unchanged

- `BuildSectorGeneratedGeometry()`, `BuildSectorMeshes()`, and `SectorMeshPreview::RebuildRendererResources()` can likely keep consuming a derived, already-valid `SectorTopologyMap` unchanged.
- `SectorCollisionWorld::BuildFromTopology()` and gameplay preview movement can remain unchanged if derivation produces valid one-sided/two-sided linedefs, sidedefs, sectors, heights, and portal flags.
- Lightmap layout/bake/status/hash can remain mostly unchanged if derived topology preserves stable IDs and all geometry-affecting fields. The lightmap source hash already captures the runtime topology contract.
- 3D surface picking/highlighting can remain unchanged if generated surfaces continue to carry topology IDs that map back to selected sectors/sidedefs.

## Which Consumers Would Need Adaptation

- The bridge into runtime systems would need a derivation/validation step from any future permissive authoring graph into `SectorTopologyMap`. Runtime consumers should not be pointed directly at loose graph state.
- 3D selection and material editing need an identity mapping if derived sector/sidedef IDs can change after graph edits. Current selection refs are topology IDs, not authoring graph IDs.
- Lightmap stale detection and reuse require stable derived IDs and stable generated surface ordering. If re-derivation changes IDs for the same logical floor/wall, existing baked metadata becomes stale even when geometry looks unchanged.
- Any future authoring graph metadata for portals, doors, or linedef specials must be copied onto the derived linedef/sidedef structures before collision, preview, or selection consumers can see it.
- Texture import/use can stay map-level, but derivation must preserve texture IDs referenced by sectors/sidedefs and avoid introducing references missing from `texturesById`.

## Risks / Unknowns

- The largest risk is ID churn during derivation. Mesh selection, material editing, lightmap self-hit filtering, and source-hash stability all depend on sector/linedef/sidedef IDs being stable enough to represent the same logical surface.
- Generated surface order is implicit. Lightmap layouts compare surface count and rely on matching generated geometry; a derivation pass that changes sector/sidedef ordering may invalidate otherwise reusable lightmaps.
- `ceilingSky` affects both generated geometry and baked-light eligibility. A future authoring graph must preserve it as derived sector data, while map-level sky visual settings should remain preview-only.
- Portal semantics are currently inferred from two-sided linedefs plus `blocksPlayer`, floor/ceiling deltas, and sector heights. Future door/action metadata would need a clear place in derived topology before runtime consumers can make use of it.
- Middle texture behavior is specialized: it receives baked lighting, is alpha-tested in preview, and is not treated as a same-logical-surface occluder. That behavior should be preserved unless explicitly changed.

## Recommended Follow-Up Questions

- What stable identity should a derived sector, sidedef, and generated surface use when the authoring graph is edited but the logical room/wall remains the same?
- Should lightmap reuse be based strictly on current topology IDs, or should a future derivation layer provide persistent logical surface IDs?
- Where should future portal/action metadata live so collision, preview selection, and serialization can consume it without making loose authoring lines part of runtime topology?
- Should 3D preview continue to show only the last valid derived topology while the authoring graph is invalid?
- Which editor actions should trigger derivation immediately, and which should allow invalid authoring state until an explicit validate/build step?
